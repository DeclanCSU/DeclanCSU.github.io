Diagnostic Methodology

Regardless of the domain, it is true that the process for isolating a fault follows the same structure. Thus it is also true that skipping steps is how bugs get "fixed" in a way that just moves them somewhere less visible rather than actually solving them.

  1. Reproduce reliably: a fault that you cannot reproduce consistently cannot truly be verified as fixed
  2. Isolate the scope: determine whether the fault is in the system or the data, the environment or their interaction
  3. Form a falsifiable hypothesis: commit to a specific claim about the cause so that it can actually be tested and confirmed
  4. Change one variable at a time: multiple simultaneous changes to make the results unattributable
  5. Document findings: record what has been tried and what has happened. This includes failed attempts
  6. Verify the fix: confirm the resolution in the same conditions where the fault had originally appeared

Performance Impact

Algorithm selection is a problem solving decision that brings about direct performance consequences. In a game that is running at 60 fps, per-frame operations have a budget of roughly 16ms. Choosing the wrong data structure for a collision query or for pathfinding lookup can make that budget rather impossible to meet.

Complexity Name Games example Scales to 10,000 entities?
O(1) Constant Hash map inventory with lookup by item ID Yes
O(log n) Logarithmic Binary search on a sorted leaderboard Yes
O(n) Linear Iterating all entities to apply gravity Yes, with care
O(n log n) Linearithmic Sorting entities by draw order each frame Marginal
O(n²) Quadratic Naive collision check: every entity vs every other No, use a spatial hash or BVH
