Communication Skills

Technical clarity, structured documentation, as well as the ability to translate complex ideas for any audience.

Communication in a games studio

Games are built by teams that work with cross discipline and different areas of experience. In this regard, a developer who cannot write a clear bug report or contribute to a design review will make progress difficult. They need flexibility, they need to be able to explain a technical constraint to an artist that is effective regardless of how good their code quality may in fact be. In a small independent studio, every miscommunication can have a direct cost in terms of rework.

My communication experience itself spans technical documentation at Datacom, my previous place of employment, as well as academic report writing at CSU. I also have customer facing support under pressure from Coles expereince. All of these are contexts where clarity and structure under time constraints matter very significantly.

Technical documentation: Incident report format

Example Structure – Account Lockout Incident

Real Example

Below is an example of the incident report format that I had used at Datacom. This structure ensures that the next person to read it can actually act on this report immediately without needing a verbal briefing to proceed properly.

Field Content
Incident ID INC-2025-0412
Reported 2025-04-12 09:14 AEST
Severity Medium, single user affected, no data loss
Summary User account locked out after having 10 failed login attempts. Stale cached credentials on shared workstation were identified as root cause.
Steps Taken Unlocked account in AD. Cleared cached credentials on the workstation. Confirmed successful login. Advised user on password manager use.
Resolution Resolved at 09:38 AEST. No escalation/transfer to second level team was required.
Follow-up Flagged shared workstation for group policy review to enforce credential refresh interval.

Bug reporting for games development

What a useful bug report contains

Studio workflow

A vague bug report such as "game crashes sometimes" is nearly worthless when given to a developer. Whereas a precise one will eliminate hours of reproduction work trying to emulate the issue before any fix can actually begin.

Section What to Include Bad Example Good Example
Title Specific, searchable summary "Game broken" "Player falls through floor on Level 3 spawn platform"
Steps to Reproduce Exact numbered sequence "Load the level and jump" "1. Load Level 3. 2. Spawn at checkpoint B. 3. Jump toward the left platform edge."
Expected Behaviour What should happen "It should work" "Player lands on platform surface and retains normal movement."
Actual Behaviour What actually happens "It crashes" "Player clips through the collider and falls into the void, triggering a respawn."
Environment Build, OS, hardware "My PC" "Build v0.4.2, Windows 11, RTX 3070, 1440p, fullscreen."

Communication Modes

Written Technical

Incident reports, runbooks, project reports. Structured for a reader who needs to act on the content without asking follow-up questions.

Cross-discipline

Explaining a technical constraint to a non-technical audience. Translating between developer and designer vocabulary without losing the important detail.

Customer-facing

IT support under pressure. Remaining clear and helpful to a frustrated user while simultaneously diagnosing the underlying problem.

Collaborative

Flagging blockers early, being explicit about assumptions, and updating shared documentation so teammates can work without waiting for a verbal handover.

In Summary

The incident report and bug report formats above are examples of communication as a technical discipline. The format is not bureaucracy, it is a tool that makes the information immediately usable by whoever receives it. At Datacom I wrote and revised these documents under live incident conditions until the structure became automatic.

That same discipline transfers directly to games development. A clear design document, a well-structured code review comment, or a concise stand-up update all operate on the same principle: the reader should be able to act on the information without having to ask what you meant.
